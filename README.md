# Multi-LLM Council v3.0

[![Claude Code Plugin](https://img.shields.io/badge/Claude%20Code-Plugin-blueviolet)](https://github.com/hadamyeedady12-dev/multi-llm-council)
[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](https://opensource.org/licenses/MIT)

> 다중 LLM CLI를 활용한 집단 지성 협의 시스템

Claude Code가 의장으로서 Gemini CLI, Codex CLI를 조율하여 **추가 API 비용 없이** 통합 협의 프로세스를 진행합니다.

## 빠른 시작

```
/영끌 <질문>
```

**단 하나의 명령어로 3개 LLM의 집단 지성을 활용합니다.**

## 특징

| 특징 | 설명 |
|------|------|
| **무비용** | 기존 CLI 구독만으로 운영 (추가 API 비용 없음) |
| **집단 지성** | 3개 LLM의 역할 기반 분석 및 교차 검증 |
| **명시적 평가** | 위원 이름을 명시한 투명한 평가 시스템 |
| **병렬 실행** | 3개 위원 동시 호출로 빠른 응답 |

## 설치

### 1. CLI 도구 설치

```bash
# Gemini CLI
brew install gemini-cli
gemini auth login

# Codex CLI
npm install -g @openai/codex
codex login

# Claude Code
npm install -g @anthropic-ai/claude-code
```

### 2. 플러그인 설치

**방법 1: 마켓플레이스 (권장)**
```
/plugin marketplace add https://github.com/hadamyeedady12-dev/multi-llm-council
/plugin install council
```

**방법 2: 직접 클론**
```bash
cd ~/.claude/plugins
git clone https://github.com/hadamyeedady12-dev/multi-llm-council.git
```

설치 후 Claude Code를 재시작하세요.

## 동작 방식

### 3단계 협의 프로세스

```
┌─────────────────────────────────────────────────────────────┐
│                   /영끌 실행 흐름                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Stage 1: 병렬 의견 수집                                     │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐                     │
│  │ Gemini  │  │ Claude  │  │ Codex   │  ← 동시 호출         │
│  └────┬────┘  └────┬────┘  └────┬────┘                     │
│       │            │            │                           │
│       ▼            ▼            ▼                           │
│  Stage 2: 명시적 교차 평가                                   │
│  • Gemini가 Codex, Claude 평가                              │
│  • Codex가 Gemini, Claude 평가                              │
│  • Claude가 Gemini, Codex 평가                              │
│       │            │            │                           │
│       └────────────┼────────────┘                           │
│                    ▼                                        │
│  Stage 3: 최종 종합 (Claude 의장)                            │
│  • 합의점 도출 및 이견 명시                                  │
│  • 우선순위별 권장사항                                       │
│  • 신뢰도 계산                                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 평가 기준 (100점 만점)

| 항목 | 비중 | 설명 |
|------|------|------|
| 정확성 | 30% | 사실 정확성, 검증 가능성 |
| 깊이 | 25% | 근본 원인 분석, 다각적 관점 |
| 실용성 | 25% | 적용 가능성, 구체적 예시 |
| 명료성 | 20% | 논리 구조, 간결함 |

### 신뢰도 계산

```
신뢰도 = (합의율 × 0.4) + (평균점수/100 × 0.3) + (근거품질 × 0.3)

등급:
• 0.8+: 높음 (신뢰 가능)
• 0.5-0.8: 중간 (참고용)
• 0.5 미만: 낮음 (추가 검토 권장)
```

## 사용 사례

### 언제 /영끌을 사용하면 좋을까요?

| 상황 | 예시 질문 |
|------|----------|
| **기술 선택** | "Python vs Go, 백엔드에 어떤 언어가 적합할까?" |
| **아키텍처 결정** | "모놀리식 vs 마이크로서비스, 우리 팀에 맞는 구조는?" |
| **디버깅 협업** | "이 에러의 근본 원인과 해결 방안은?" |
| **코드 리뷰** | "이 설계 패턴의 장단점과 개선 방향은?" |
| **UX 검토** | "이 UI 흐름의 사용성 문제점과 개선안은?" |

### 디버깅 협업 예시

복잡한 버그를 해결할 때 각 위원의 강점이 시너지를 발휘합니다:

```
버그 발생
    ↓
[Codex] 에러 로그 분석 → 루트 원인 파악 → 패치 제안
    ↓
[Claude] 수정안 논리 검토 → 부작용/보안 이슈 확인
    ↓
[Gemini] 최신 라이브러리 확인 → Best Practice 검색
    ↓
최종 수정 적용
```

## 출력 예시

```markdown
# LLM 위원회 결론

## 질문
Python과 Go 중 어떤 언어가 백엔드에 더 적합한가?

## Stage 1: 전문가별 분석

### 탐색/UX 전문가 (Gemini)
**UX 관점:** 개발자 경험(DX) 분석
**최신 트렌드:** 현재 업계 동향 검색 결과

### 구현/디버깅 전문가 (Codex)
**구현:** Python이 빠른 개발에 유리
**디버깅:** Go가 에러 추적에 용이

### 설계/검증 전문가 (Claude)
**확장성:** 마이크로서비스는 Go 적합
**아키텍처:** 상황별 최적 선택 제시

## Stage 2: 명시적 교차 평가
| 위원 | 정확성 | 깊이 | 실용성 | 명료성 | 총점 |
|------|--------|------|--------|--------|------|
| Gemini | 28/30 | 22/25 | 23/25 | 18/20 | 91 |
| Codex | 27/30 | 24/25 | 22/25 | 19/20 | 92 |
| Claude | 29/30 | 23/25 | 24/25 | 18/20 | 94 |

## Stage 3: 최종 결론

### 권장사항
1. **즉시 실행:** 프로젝트 요구사항 명확화
2. **단기 개선:** 팀 기술 스택 고려
3. **장기 고려:** 확장성 계획 수립

### 신뢰도
점수: 0.85 | 등급: 높음

---
*이 결론은 Gemini, Codex, Claude의 집단 지성을 종합한 것입니다.*
```

## 에러 처리

| 상황 | 처리 방식 |
|------|----------|
| CLI 실패 | Exponential Backoff 2회 재시도 |
| 위원 1명 실패 | 2인 위원회로 진행 |
| 2명 미만 응답 | 세션 중단, 사용자 알림 |
| JSON 파싱 실패 | 자유 형식으로 폴백 |
| 타임아웃 | 120초/위원 |

## 핵심 원칙

1. **독립성** - 각 위원은 역할 기반으로 독립적으로 분석
2. **명시성** - 평가 시 위원 이름을 명확히 표시
3. **투명성** - 모든 과정과 근거 공개
4. **합의 지향** - 다수결보다 논거의 질 중시
